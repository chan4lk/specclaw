# Design: specclaw pr — PR Creation Command

**Change:** pr-command
**Created:** 2026-05-13

## Architecture

Follows the existing specclaw script pattern: thin bash scripts, no external dependencies beyond `gh` CLI and standard POSIX tools. Each script is self-contained with its own `yaml_val` helper (copy from gh-sync.sh — this duplication is intentional for script portability).

## File Changes Map

| File | Action | Description |
|------|--------|-------------|
| `skill/scripts/validate-change.sh` | Modify | Add `pr` case to the phase switch |
| `skill/scripts/pr.sh` | Create | New script — test policy, PR creation |
| `skill/SKILL.md` | Modify | Add `specclaw pr` section after `specclaw verify` |
| `skill/templates/config.yaml` | Modify | Add `pr.test_policy` field with comment |

## validate-change.sh Changes

Add one new case in the `case "$PHASE" in` block:

```bash
pr)
  if [[ ! -f "$CHANGE_DIR/verify-report.md" ]]; then
    fail "Missing verify-report.md — run specclaw verify first"
  fi
  # Idempotency guard: warn if PR already created
  if [[ -f "$CHANGE_DIR/status.md" ]] && grep -q "^\*\*PR:\*\*" "$CHANGE_DIR/status.md" 2>/dev/null; then
    local pr_url
    pr_url="$(grep '^\*\*PR:\*\*' "$CHANGE_DIR/status.md" | sed 's/\*\*PR:\*\* //' | head -1)"
    echo "WARNING: PR already created: $pr_url" >&2
    exit 0
  fi
  ;;
```

## pr.sh Design

### Usage
```
pr.sh <specclaw_dir> <change_name>
```

### Flow

```
1. Parse args, locate config.yaml, change dir
2. validate-change.sh pr phase (fail fast)
3. check_test_policy()   — prompt if missing, save to config
4. enforce_test_policy() — check test evidence if policy != none
5. build_pr_body()       — assemble markdown from artifacts
6. build_pr_title()      — extract summary from proposal.md
7. gh pr create          — run with title + body
8. save_pr_url()         — append to status.md
```

### Key Functions

**`check_test_policy()`**
```bash
# Read pr.test_policy from config
# If missing: loop prompt until valid answer (none|unit|e2e|both)
# Write pr: section to config.yaml using append + sed approach:
#   If `pr:` section exists → update test_policy line
#   If not → append pr:\n  test_policy: "<answer>" to config
```

**`yaml_set_pr_policy()`**
```bash
# Appending strategy (safe — no full-file rewrite):
# Check if `^pr:` exists in config. If yes, use sed to replace/insert test_policy line.
# If no, append two lines: "pr:" and "  test_policy: \"$policy\""
```

**`enforce_test_policy()`**
```bash
# policy=none → return 0
# Check build.test_command non-empty
# Check verify-report.md for test keywords: test|passed|failed|coverage|e2e|unit|assert
# Failures use same FAILURES[] + fail() pattern as validate-change.sh
# Respects $STRICT for exit behavior
```

**`build_pr_body()`**
```bash
# 1. Summary: grep proposal.md for lines after "## Problem" or "## Proposed Solution", take first 15 lines
# 2. Acceptance Criteria: extract "## Acceptance Criteria" section from spec.md (up to next ##)
#    Fallback: first 40 lines of spec.md
# 3. Verification: grep verify-report.md for verdict line (PASS|FAIL|PARTIAL), take first 20 lines
# 4. Tests: policy value + grep verify-report.md for test evidence lines (up to 10 lines)
#    Omit section entirely if policy=none
# 5. Closes: if github.sync=true, extract #N from status.md
# 6. Footer: "🤖 Generated by SpecClaw"
```

**`build_pr_title()`**
```bash
# Scan proposal.md for first non-blank, non-header, non-bold line
# Format: "[specclaw] <change_name>: <summary>"
# Truncate to 72 chars
```

### Config Update Strategy

Avoid full-file rewrite. Two cases:

1. `pr:` section exists in config.yaml → use `sed` to insert/replace `test_policy` line under it.
2. `pr:` section absent → append to end of file:
```yaml

# PR creation settings
pr:
  test_policy: "none"   # none | unit | e2e | both — set on first specclaw pr run
```

This is safe because `yaml_val` reads top-level and one-level nested keys.

### PR Body Template

```markdown
## Summary
<proposal problem + solution excerpt>

## Acceptance Criteria
<from spec.md ## Acceptance Criteria section>

## Verification
**Verdict:** PASS

<verify-report first 20 lines>

## Tests
**Policy:** unit

<test evidence lines from verify-report>

---
Closes #9

🤖 Generated by SpecClaw
```

## SKILL.md Changes

Insert new section after `### specclaw verify <change>` block (after the Remediation subsection), before `### specclaw status`:

```markdown
### `specclaw pr <change>`
**Trigger:** "specclaw pr", "create PR", "open pull request"

1. **Validate:** Run `bash skill/scripts/validate-change.sh .specclaw <change> pr`. Requires verify-report.md. Exits if PR already exists.
2. **Test policy:** Run `bash skill/scripts/pr.sh <specclaw_dir> <change_name>` which handles first-run prompt and enforcement internally.
3. The script creates the PR and saves the URL. Report the PR URL to the user.
```

## templates/config.yaml Changes

Add at end of file (after `automation` section):

```yaml

# PR creation settings
pr:
  test_policy: ""   # Set on first `specclaw pr` run. Options: none | unit | e2e | both
                    # none  — no test enforcement
                    # unit  — requires unit test evidence in verify-report + build.test_command set
                    # e2e   — requires e2e test evidence in verify-report + build.test_command set
                    # both  — requires unit + e2e evidence
```

## Error Handling

All errors follow existing pattern:
- `die()` for unrecoverable (missing gh, bad args)
- `fail()` collected into FAILURES array, reported at end, exit behavior controlled by `$STRICT`
