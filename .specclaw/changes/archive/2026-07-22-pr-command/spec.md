# Spec: specclaw pr — PR Creation Command

**Change:** pr-command
**Created:** 2026-05-13

## Functional Requirements

### FR1 — Phase Validation
`specclaw pr <change>` must refuse to proceed unless `verify-report.md` exists for the change. This is enforced via `validate-change.sh .specclaw <change> pr`. In strict mode, missing `verify-report.md` exits 1 with a clear error. In non-strict mode, emits a warning but continues.

### FR2 — Test Policy: First-Run Prompt
On first invocation (when `pr.test_policy` key is absent from `config.yaml`), `pr.sh` must:
- Print a prompt asking: "Do you plan to implement automated tests for this project? Options: none / unit / e2e / both"
- Read the user's answer from stdin
- Validate the answer is one of: `none`, `unit`, `e2e`, `both`
- Persist the answer to `config.yaml` under a `pr:` section as `test_policy: <answer>`
- Subsequent runs must skip the prompt and read from config.

### FR3 — Test Policy: Enforcement
When `pr.test_policy` is `unit`, `e2e`, or `both`:
- `build.test_command` must be non-empty in `config.yaml`. If empty → fail (strict) or warn (non-strict).
- `verify-report.md` must contain evidence of tests running (grep for keywords: "test", "passed", "failed", "coverage", "e2e", "unit"). If no test evidence → fail (strict) or warn (non-strict).
- When policy is `none`, skip enforcement entirely.

### FR4 — PR Body Auto-Population
`pr.sh` must build the PR body from:
- **Summary:** Problem + solution lines from `proposal.md`
- **Acceptance Criteria:** Extracted from `spec.md` (the `## Acceptance Criteria` section, or first 40 lines if no such section)
- **Verification:** Verdict (PASS/FAIL/PARTIAL) + first 20 lines of `verify-report.md`
- **Tests:** Policy value + test evidence excerpt from `verify-report.md` (if policy != none)
- **Closes #N:** If `github.sync: true` and issue number found in `status.md`
- **Footer:** `🤖 Generated by SpecClaw`

### FR5 — PR Title
Title format: `[specclaw] <change-name>: <one-line summary>` where summary is the first non-blank, non-header line from `proposal.md`. Falls back to `[specclaw] <change-name>` if extraction fails.

### FR6 — PR Creation
Run `gh pr create --base main --title "..." --body "..."`. Authentication via `gh` CLI (already authed in environment).

### FR7 — Save PR URL
After successful creation, append `**PR:** <url>` to `status.md`.

### FR8 — GitHub Issue Link
If `github.sync: true` in config and issue number found in `status.md`, include `Closes #N` in the PR body footer.

## Acceptance Criteria

- AC1: Running `specclaw pr <change>` when `verify-report.md` is absent → exits with "Missing verify-report.md — run specclaw verify first" (strict) or a warning (non-strict).
- AC2: On first run with no `pr.test_policy` in config → prompts user, accepts `none|unit|e2e|both`, rejects invalid input and re-prompts, writes chosen value to config.yaml.
- AC3: On subsequent runs → no prompt; reads `pr.test_policy` from config silently.
- AC4: With `test_policy: unit` and empty `build.test_command` → fails (strict) / warns (non-strict) before creating PR.
- AC5: With `test_policy: unit` and verify-report has no test evidence → fails (strict) / warns (non-strict).
- AC6: With `test_policy: none` → no test checks run, PR created immediately after validation.
- AC7: PR body contains all four sections: Summary, Acceptance Criteria, Verification, Tests.
- AC8: PR body contains `Closes #N` when `github.sync: true` and issue number is in `status.md`.
- AC9: PR body does NOT contain `Closes #N` when `github.sync: false` or no issue found.
- AC10: After PR creation, `status.md` is updated with the PR URL on a `**PR:**` line.
- AC11: Running `specclaw pr` when a PR already exists (PR URL already in status.md) → warns "PR already created: <url>" and exits 0 without creating a duplicate.

## Edge Cases

- `verify-report.md` exists but is empty → treated as no test evidence (FR3 enforcement applies)
- `spec.md` has no `## Acceptance Criteria` section → use first 40 lines of spec.md
- `proposal.md` missing (unlikely since it's required earlier) → PR body summary section says "See change: `<change-name>`"
- `gh` CLI not available → exit with "gh CLI required for PR creation"
- PR creation fails (gh exits non-zero) → report error, do NOT write PR URL to status.md
