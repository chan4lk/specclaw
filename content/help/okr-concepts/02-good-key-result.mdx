# What Makes a Good Key Result?

A **Key Result** (KR) answers: *"How will we know if we're getting there?"* It's a measurable outcome — not a task or activity.

<Callout type="warning">
Key Results measure **outcomes**, not outputs. "Ship 5 features" is an output. "Increase weekly active users from 10k to 50k" is an outcome.
</Callout>

## The KR Formula

A well-formed Key Result follows this pattern:

> **Verb** + **metric** + **from X to Y** + **by [date]**

**Examples:**
- Increase Net Promoter Score from 32 to 50 by end of Q2
- Reduce average onboarding time from 14 days to 5 days
- Grow enterprise accounts from 12 to 40

## What Makes a KR "Good"?

| Criterion | What It Means |
|-----------|--------------|
| **Measurable** | You can objectively score it 0.0–1.0 |
| **Outcome-based** | Focuses on what changes, not what you do |
| **Ambitious** | A score of 0.7 at cycle end is success |
| **Controllable** | The team has meaningful influence over it |
| **Finite** | Has a clear end-state |

## KRs vs. Tasks

| Key Result | Task (not a KR) |
|-----------|-----------------|
| Reduce bug backlog from 200 to 50 | Fix 10 bugs per week |
| Achieve 99.9% uptime | Deploy monitoring infrastructure |
| Increase mobile conversion rate to 4% | Redesign checkout flow |

<Callout type="tip">
Aim for 2–5 Key Results per Objective. Too few and the path is unclear; too many and you're listing tasks again.
</Callout>

## The "How Would We Achieve This Unethically?" Test

If a KR can be gamed easily (e.g., inflate the metric without real value), it's poorly designed. Apply this test: *"Could we hit this number without actually achieving the Objective?"* If yes — rewrite the KR.
